架构分层

Pi 采用清晰的分层架构，每层职责单一，上层依赖下层：

T
pi-tui
终端 UI 库 · 差分渲染 · 高性能文本界面
独立层
↓ 被 coding-agent 依赖
C
pi-coding-agent
编码代理 CLI · 工具系统 · 会话管理 · 最终产品
依赖 agent + ai + tui
↓ 依赖 agent 运行时
A
pi-agent-core
Agent 运行时 · 传输抽象 · 状态管理 · 工具调用
依赖 ai
↓ 依赖 LLM API
L
pi-ai
统一 LLM API · 多 Provider · 模型发现 · 最底层
被所有上层消费
TUI 层详细说明 ▶

核心能力：差分渲染（Differential Rendering）— 只重绘终端中发生变化的部分，而非全屏刷新。

技术特点：

  • 包含原生预编译二进制（native/win32/ 和 native/darwin/）
  • 使用 node --test 进行测试（与其他包的 vitest 不同）
  • 依赖 get-east-asian-width 处理东亚字符宽度
  • 依赖 marked 进行 Markdown 渲染

关键问题：差分渲染的具体算法实现？

Coding Agent 层详细说明 ▶

核心能力：面向开发者的交互式编码助手，是最终用户直接使用的 CLI 产品。

四大工具：

工具功能对应文件
read读取文件内容src/core/tools/read.ts
bash执行 shell 命令src/core/tools/bash.ts
edit精确文本替换编辑src/core/tools/edit.ts
write写入/创建文件src/core/tools/write.ts

其他能力：

  • 会话管理（session）— 持久化对话历史
  • 交互模式（TUI）和 RPC 模式
  • HTML 导出功能
  • 主题系统（src/modes/interactive/theme/）
  • 扩展机制（examples/extensions/）
Agent Core 层详细说明 ▶

核心能力：通用 Agent 框架，不绑定具体场景，可被其他类型的 Agent 复用。

关键抽象：

  • Transport — 传输层抽象，解耦通信方式
  • State Management — 状态管理，维护 Agent 运行状态
  • Attachment Support — 附件处理（图片、文件等）
  • Tool Orchestration — 工具调用编排

导出路径：主入口 + ./node 子路径

AI 层详细说明 ▶

核心能力：屏蔽不同 LLM Provider 的差异，提供统一的调用接口。

支持的 Provider：

OpenAI Anthropic Google Gemini Google Vertex Mistral Azure OpenAI AWS Bedrock

关键文件：

  • src/providers/*.ts — 各 Provider 实现
  • src/models.generated.ts — 自动生成的模型元数据 禁止手动修改
  • scripts/generate-models.ts — 模型生成脚本

导出设计：主入口 + 各 Provider 子路径（./anthropic, ./openai-responses 等），支持按需导入。

TypeScript 策略

关键配置

tsconfig.base.json 
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    
    // 核心约束：只使用可擦除语法
    "erasableSyntaxOnly": true,
    
    // 直接导入 .ts 文件，构建时自动改写
    "allowImportingTsExtensions": true,
    "rewriteRelativeImportExtensions": true,
    
    // 装饰器支持
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true,
    "useDefineForClassFields": false
  }
}

erasableSyntaxOnly 约束

这是 Node.js strip-only mode 的要求，构建时直接去掉类型注解即可，不需要 emit 成 JS。

禁止使用的语法替代方案
enum对象常量 + as const
namespace / moduleES Module
import = / export =标准 ESM import/export
参数属性（Parameter Properties）显式字段 + 构造函数赋值

构建体系

构建顺序（拓扑排序）

package.json scripts 
"build": "
  cd packages/tui && npm run build &&
  cd packages/ai && npm run build &&
  cd packages/agent && npm run build &&
  cd packages/coding-agent && npm run build
"

顺序很重要：tui → ai → agent → coding-agent

验证相对导入路径正确性
4. Shrinkwrap 检查

5. 类型检查
tsgo --noEmit
6. 浏览器兼容性

供应链安全

这是 Pi 项目非常突出的工程实践，值得深入学习：

精确版本锁定

SEC

所有外部依赖必须 pinned，.npmrc 设置 save-exact=true

Lockfile 保护

SEC

pre-commit 钩子阻止意外提交 lockfile 变更，需 PI_ALLOW_LOCKFILE_CHANGE=1

Shrinkwrap 发布

SEC

coding-agent 发布时包含 npm-shrinkwrap.json，锁定传递依赖

生命周期审查

SEC

shrinkwrap 生成脚本有 allowlist，新生命周期脚本需人工审查

推荐学习路线

Phase 1: TUI 层（独立，无依赖）
理解差分渲染算法，这是 Pi 的性能基础。
分支: study/tui · 重点: src/ 渲染逻辑
Phase 2: AI 层（被所有上层消费）
理解 Provider 注册机制，看一个 Provider（如 OpenAI）的完整实现。
分支: study/ai · 重点: src/providers/, 模型发现系统
Phase 3: Agent 层（运行时框架）
理解 Transport 抽象、状态管理、工具调用生命周期。
分支: study/agent · 重点: src/transport/, src/state/
Phase 4: Coding Agent 层（最终产品）
理解工具实现、会话管理、CLI 入口、扩展机制。
分支: study/coding-agent · 重点: src/core/tools/, src/modes/

请求追踪示例

一个完整的用户请求在 Pi 中的流转路径：

flow 
用户输入
  ↓
[coding-agent] CLI 入口 (src/cli.ts)
  ↓ 解析参数，启动会话
[coding-agent] 模式选择 (TUI / RPC / One-shot)
  ↓
[coding-agent] 工具系统 (src/core/tools/)
  ↓ 构建 Tool Schema，发送给 LLM
[agent] Agent 运行时 (src/)
  ↓ 状态管理，传输封装
[ai] Provider 调用 (src/providers/)
  ↓ HTTP 请求
[外部] OpenAI / Anthropic / Google API
  ↓ 返回结果
[ai] 统一响应格式
  ↓
[agent] 解析工具调用
  ↓
[coding-agent] 执行工具 (read/bash/edit/write)
  ↓ 结果回传
[ai] 发送工具结果给 LLM
  ↓ 最终回复
[coding-agent] 渲染输出
  ↓
[tui] 差分渲染到终端

待解问题

在学习过程中记录下的疑问，逐个击破：

tsgo 是什么？和 tsc 的关系？ ▶

待研究。从使用方式看，它替代了 tsc 作为类型检查和构建工具，可能是一个更快的 TypeScript 编译器或 Go 重写的替代品。

typebox 在项目中扮演什么角色？ ▶

多个包都依赖 typebox（1.1.38）。从名字推断，可能用于运行时类型验证和 JSON Schema 生成，在 Tool Schema 定义中起关键作用。

差分渲染的具体算法实现？ ▶

TUI 的核心竞争力。需要深入 packages/tui/src/ 查看具体实现，理解如何计算终端屏幕的最小变更集。

Provider 的懒加载注册机制？ ▶

AI 包支持多个 Provider，但用户可能只使用其中一个。需要理解如何实现按需加载，避免启动时加载所有 Provider 依赖。

扩展系统的加载机制（jiti 动态加载？） ▶

coding-agent 依赖 jiti（2.7.0），这是一个 TypeScript 运行时加载器。需要理解扩展是如何在运行时被动态加载和注册的。
